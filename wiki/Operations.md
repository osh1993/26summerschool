# 운영자 게시 절차

이 문서는 공개 저장소에 둘 수 있는 운영 절차만 설명합니다. 실제 참가자 이름, 연락처, Google Sheet 주소, Apps Script 관리 URL, 토큰을 기록하지 마세요.

## 게시 전

1. Google Forms 원본 응답을 운영 시트에 동기화합니다.
2. 조편성, 이동수요와 차량 배치를 운영자가 검토합니다.
3. 중복 탑승, 정원 초과, 시간 충돌, 미등록 차량을 검사합니다.
4. 공개 코드 누락과 공개 동의를 확인합니다.
5. 공개 JSON에서 실명·전화번호·이메일·생년월일·상세주소가 나오지 않는지 검사합니다.
6. 차단 오류가 0일 때만 새 공시본을 게시합니다.

## Pages 게시 확인

1. [공개 안내 페이지](https://26summerschool.vercel.app/)를 휴대전화에서 엽니다.
2. 화면 상단이 `최신 공시 정보`인지 확인합니다.
3. 마지막 공시 시각과 공시본 번호를 확인합니다.
4. 날짜·방향 필터와 공개 코드 검색을 시험합니다.
5. 전체 정원은 운전자 포함, 남은 좌석은 승객 좌석으로 표시되는지 확인합니다.
6. 360px 폭에서 가로 스크롤 없이 보이는지 확인합니다.

## 데이터 연결 실패 시

안내 페이지의 데이터 사용 순서는 다음과 같습니다.

1. Apps Script 공개 JSON
2. 선택적으로 생성한 마지막 정상 공시본 `latest.json`(실제 운영본을 검증해 저장한 경우만)
3. 합성 샘플 `sample.json`

`마지막 정상 공시본`이나 `샘플 데이터`가 표시되면 실제 Google 게시 URL, 배포 권한, 응답 스키마를 확인합니다. 장애 중에는 잘못된 새 공시본보다 검증된 이전 공시본을 유지합니다.

## Wiki 갱신

GitHub Wiki는 JavaScript가 실행되지 않으므로 상세 운행을 동적으로 가져오지 않습니다. Wiki에는 사용법, 비식별 집계와 Pages 링크만 유지합니다.

Wiki는 본 저장소와 별도인 `26summerschool.wiki.git` 저장소입니다. `wiki/` 폴더를 본 저장소에 push하는 것만으로 Wiki가 갱신되지는 않습니다.

1. GitHub 웹 UI에서 첫 Wiki 페이지를 만들어 Wiki 저장소를 초기화합니다.
2. `.wiki.git`을 별도로 clone합니다.
3. 이 폴더의 Markdown만 복사해 검토합니다.
4. 개인정보가 없음을 확인한 뒤 Wiki 저장소에 push합니다.

## 긴급 중단

개인정보 또는 잘못된 배치가 공개되면 Apps Script 공개를 중단하고 안전한 공시 안내로 교체합니다. Git 기록, 포크와 캐시에 남을 수 있으므로 단순 파일 삭제만으로 해결된다고 간주하지 말고 저장소 관리자에게 즉시 보고합니다.
